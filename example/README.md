# GhostKey Reference App

Minimal end-to-end demo of the GhostKey SDK. Shows the full flow:

1. **Login** with email (creates account on first login)
2. **Create smart account** (ERC-4337, Base Sepolia)
3. **Issue session key** — key generated in-browser via WebCrypto, only hash goes to server
4. **Send intent** — ERC-20 transfer, routed through Pimlico bundler
5. **Poll status** until confirmed

No private keys are sent to the server at any point.

---

## Prerequisites

- Node 20+
- Running GhostKey server (`make dev` from repo root)
- A Pimlico API key configured in `config.toml`

## Setup

```bash
# From repo root — start the server
cp config.toml.example config.toml
# Fill in jwt_secret + Pimlico keys in config.toml
make dev

# In another terminal — start the example app
cd example
npm install
npm run dev
```

Open http://localhost:3000.

## Environment

The app points to `http://localhost:8080` by default. Override via:

```bash
VITE_API_URL=http://your-server:8080 npm run dev
```

## What it demonstrates

| Concept | Where |
|---------|-------|
| `GhostKeyProvider` wrapping the app | `src/main.tsx` |
| `useLogin` — email auth | `src/App.tsx` |
| `useAccount` — smart account creation | `src/App.tsx` |
| `generateSessionKey()` — WebCrypto in-browser | `src/App.tsx` |
| `useSessionKey` — session key issuance | `src/App.tsx` |
| `useSendIntent` — ERC-20 transfer intent | `src/App.tsx` |

## Non-custodial proof

The private key generated by `generateSessionKey()` is shown in the UI (in amber) to prove it stays client-side. Only the SHA-256 hash (`keyHash`) is sent to the server. In production, you would not display the private key — store it in memory only.
